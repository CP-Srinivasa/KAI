# Alerting System

The AI Analyst Trading Bot sends real-time alerts and scheduled digests via **Telegram** and **Email**.
Alerts are generated by evaluating a set of configurable threshold rules against scored documents.

---

## Alert Types

| Type | Trigger | Default Channel |
|------|---------|-----------------|
| `BREAKING` | High-impact document passes all rule thresholds immediately | Telegram |
| `WATCHLIST_HIT` | A watchlist entity (BTC, ETH, SEC‚Ä¶) is mentioned in a high-credibility source | Telegram |
| `DAILY_BRIEF` | Scheduled digest of top-scoring documents for the day | Email |
| `DIGEST` | On-demand or scheduled multi-item summary | Telegram + Email |
| `ANOMALY` | (Reserved) Unusual pattern detected | TBD |

---

## Rule Configuration

Rules are defined as `AlertRule` dataclasses in `app/alerts/rules.py`.
Every field uses **AND logic**: a document must satisfy *all* active thresholds to trigger an alert.

### Fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | `str` | required | Unique rule identifier |
| `alert_type` | `AlertType` | `BREAKING` | Type of alert to send |
| `channels` | `list[AlertChannel]` | `[TELEGRAM]` | Delivery channels |
| `enabled` | `bool` | `True` | Disabled rules are never evaluated |
| `min_impact` | `float` | `0.0` | Minimum impact score (0‚Äì1) |
| `min_relevance` | `float` | `0.0` | Minimum relevance score (0‚Äì1) |
| `min_credibility` | `float` | `0.0` | Minimum credibility score (0‚Äì1) |
| `min_novelty` | `float` | `0.0` | Minimum novelty score (0‚Äì1) |
| `min_sentiment_abs` | `float` | `0.0` | Minimum absolute sentiment strength |
| `min_keyword_score` | `float` | `0.0` | Minimum keyword match score |
| `max_spam_probability` | `float` | `1.0` | Maximum allowed spam probability |
| `requires_entity_hit` | `bool` | `False` | Must match a watchlist entity |
| `requires_actionable` | `bool` | `False` | LLM must flag as actionable |
| `priority_threshold` | `DocumentPriority` | `NOISE` | Minimum priority level |
| `allowed_source_types` | `list[SourceType]` | `[]` | Restrict to specific source types (empty = all) |
| `dedup_window_hours` | `int` | `24` | Skip if same document was alerted within N hours |

### Default Rules

```python
# Breaking news ‚Äî high bar, entity required
BREAKING_ALERT_RULE = AlertRule(
    name="breaking_crypto",
    alert_type=AlertType.BREAKING,
    channels=[AlertChannel.TELEGRAM],
    min_impact=0.75,
    min_relevance=0.60,
    min_credibility=0.50,
    max_spam_probability=0.20,
    requires_entity_hit=True,
    priority_threshold=DocumentPriority.HIGH,
    dedup_window_hours=24,
)

# Watchlist hit ‚Äî lower impact bar, entity always required
WATCHLIST_HIT_RULE = AlertRule(
    name="watchlist_entity_hit",
    alert_type=AlertType.WATCHLIST_HIT,
    channels=[AlertChannel.TELEGRAM],
    min_impact=0.55,
    min_relevance=0.50,
    requires_entity_hit=True,
    max_spam_probability=0.30,
    dedup_window_hours=6,
)

# Daily digest ‚Äî permissive, runs on schedule
DAILY_DIGEST_RULE = AlertRule(
    name="daily_digest",
    alert_type=AlertType.DAILY_BRIEF,
    channels=[AlertChannel.EMAIL],
    min_impact=0.30,
    min_relevance=0.30,
    max_spam_probability=0.50,
    dedup_window_hours=48,
)
```

## Priority Levels

Priorities are ordered from lowest to highest:

```
NOISE < LOW < MEDIUM < HIGH < CRITICAL
```

`priority_threshold` uses "at least" semantics:
- Threshold `HIGH` ‚Üí only `HIGH` and `CRITICAL` pass.
- Threshold `NOISE` ‚Üí everything passes (default).

### Format

Messages use **MarkdownV2** formatting. Special characters are auto-escaped by `escape_md()`.
Maximum message length: **4096 characters** (enforced by truncation).

### Retry Behavior

Failed requests are retried with exponential backoff (tenacity):
- Max attempts: 4
- Wait: 2s ‚Üí 4s ‚Üí 8s
- Rate-limit (HTTP 429): respects `Retry-After` header


**Port 587** ‚Üí STARTTLS (recommended)
**Port 465** ‚Üí SMTPS (SSL from the start)

Emails are sent as **multipart/alternative** (plain text + HTML), ensuring compatibility with all clients.
SMTP runs in a thread-pool executor to avoid blocking the async event loop.

## Deduplication

Each alert is persisted in the `alerts` database table. Before dispatching, the system checks:

```
recently_sent(document_id, rule_name, channel, window_hours)
```

If the same document triggered the same rule on the same channel within the `dedup_window_hours` window, the alert is **silently skipped**.

Default windows:
- `breaking_crypto`: 24 hours
- `watchlist_entity_hit`: 6 hours
- `daily_digest`: 48 hours

## Dry-Run Mode

Both adapters support `dry_run=True`. In dry-run mode:
- Messages are **formatted and logged** but never sent.
- `send_text()` / `send_alert()` / `send_digest()` all return `True`.
- `healthcheck()` returns `{"healthy": True, "mode": "dry_run"}`.

Useful for CI testing and local development without credentials.

```python
adapter = TelegramAdapter(bot_token="fake", chat_id="0", dry_run=True)
await adapter.send_text("Test")  # ‚Üí True, nothing sent
```

## DigestBuilder

`DigestBuilder` filters, deduplicates, and ranks a list of `DocumentScores` for digest delivery.

```python
from app.alerts.digest import DigestBuilder

builder = DigestBuilder(
    max_items=15,        # Maximum articles in digest
    min_impact=0.30,     # Drop low-impact articles
    min_relevance=0.30,  # Drop low-relevance articles
    max_spam=0.50,       # Drop spammy articles
)

top_articles = builder.build(all_scored_documents)
```

Deduplication uses Jaccard similarity on title tokens (threshold: 0.75).
Ranking: by `DocumentPriority` (descending), then composite score.
