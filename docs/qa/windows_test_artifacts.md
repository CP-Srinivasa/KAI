# Windows Test Artifacts Documentation

This document inventories and describes the temporary test artifacts generated on Windows during local testing and development of the KAI system.

## Overview of Artifacts

When running the test suites (`python -m pytest`), code analysis (`ruff`), type checking (`mypy`), or coverage tracking locally on Windows, several files and directories are dynamically created. All of these artifacts are safely excluded from version control via `.gitignore`.

### 1. Test Databases
- **`test_e2e.db`**
  - **Location**: Repository root (`/test_e2e.db`)
  - **Description**: SQLite database automatically initialized and used by end-to-end (E2E) and integration tests to validate database migrations, repositories, and transactional behaviors on the local filesystem.
  - **Status**: Git-ignored. Safe to delete when tests are not running.

### 2. Coverage Reports
- **`.coverage`**
  - **Location**: Repository root (`/.coverage`)
  - **Description**: Binary data file generated by coverage tracking tools (e.g., `coverage.py` or `pytest-cov`) containing code execution path metrics.
  - **Status**: Git-ignored. Safe to delete.

### 3. Tool Caches (Performance Optimization)
- **`.pytest_cache/`**
  - **Location**: Repository root (`/.pytest_cache/`)
  - **Description**: Pytest internal cache containing state information (e.g., failed tests tracker for `--lf`) to speed up subsequent test executions.
  - **Status**: Git-ignored. Safe to delete.
- **`.ruff_cache/`**
  - **Location**: Repository root (`/.ruff_cache/`)
  - **Description**: Ruff cache directory storing structural metadata of checked files to optimize incremental lint runs.
  - **Status**: Git-ignored. Safe to delete.
- **`.mypy_cache/`**
  - **Location**: Repository root (`/.mypy_cache/`)
  - **Description**: Mypy type-checker cache directory storing parsed type signatures of modules.
  - **Status**: Git-ignored. Safe to delete.
- **`.hypothesis/`**
  - **Location**: Repository root (`/.hypothesis/`)
  - **Description**: Cache directory used by the Hypothesis library for property-based tests (e.g. to remember failing examples across runs).
  - **Status**: Git-ignored. Safe to delete.

## Encoding and Character Sets (cp1252 vs. UTF-8)

When developing or running tests/scripts on Windows, console outputs and file operations may default to the system's locale code page (typically **cp1252** on Western European systems) rather than **UTF-8**. This causes `UnicodeEncodeError` or `UnicodeDecodeError` when printing mathematical or non-standard Unicode characters (such as `∧` or `≥`) to the console.

To ensure compatibility across Windows development workstations and the Pi production server:
1. **ASCII Fallbacks**: Avoid using non-ASCII mathematical symbols (e.g. `∧`, `∨`, `≥`, `≤`) in direct shell outputs. Use standard equivalents (`AND`, `OR`, `>=`, `<=`). This is the pattern implemented in `scripts/eligibility_probe.py` (PR #184).
2. **Explicit Encodings**: Always specify `encoding="utf-8"` when reading or writing files via `open()` or `Path.read_text()` / `Path.write_text()`.
3. **Shell Compatibility**: Do not assume the developer's shell can render complex Unicode symbols. Terminal outputs must remain clean under standard cp1252.

## Maintenance and Hygiene

All the above-listed directories and files contain only transient data. If you experience locking issues on Windows (e.g. file locks on `test_e2e.db` or cache conflicts), run:

```powershell
# Clean up temporary caches and test databases on Windows (PowerShell)
Remove-Item -Recurse -Force .pytest_cache, .ruff_cache, .mypy_cache, .hypothesis, .coverage, test_e2e.db -ErrorAction SilentlyContinue
```
Or in Git-Bash:
```bash
rm -rf .pytest_cache .ruff_cache .mypy_cache .hypothesis .coverage test_e2e.db
```

This documentation is maintained in compliance with the **Worktree-Hygiene-Manifest** (`cleanup_manifest.md`).
