# Premium Dashboard Truth Checklist & Layout Rules

This checklist defines the styling, structural, and behavioral invariants for the Premium Signal Trail, Heatmaps, Portfolio, and Signal Quality screens. These rules ensure that dashboard UI components present an honest, accurate, and non-redundant representation of execution data.

---

## 1. Metric Calculations & De-duplication

- [ ] **No Double Counting (Raw + Approved):**
  - Raw envelopes (from `telegram_premium_channel`) and approved envelopes (ending in `_approved`) must be treated as a *single* operational signal.
  - The dashboard matrix must group envelopes by their core identifier (`origin_envelope_id` or `envelope_id` fallback) to prevent inflated signal counts.
- [ ] **Shadow vs. Approved Separation:**
  - The dashboard matrix and summaries must never aggregate Shadow signals (autonomous strategy candidates) and Approved signals (operator-confirmed premium signals) into a single metric.
  - Shadow signals must be visualised separately and never counted as trading successes or execution failures in premium statistics.
- [ ] **Signal Quality Filtering:**
  - The Signal Quality module must not penalise the system for "missing" Shadow signals.
  - Quality metrics must only evaluate signals that were approved,allowlisted, and executable under the active risk boundaries.

---

## 2. UI Status Labels & Headline Rules

- [ ] **Operative Terminal State vs. Entry Mode:**
  - The UI must *never* show generic `entry_mode` (e.g., "disabled") as the primary cause headline if the signal failed validation or risk checks first.
  - If a signal was rejected by a risk limit, it must be labeled **RISK_REJECTED** (not ENTRY_DISABLED).
  - If execution was globally disabled, it must show **ENTRY_DISABLED** (status derived from `rejected_entry_mode`).
- [ ] **External Signals Execution State:**
  - External Signals panels must never display "accepted" envelopes as "executed".
  - A signal is "executed" only when it reaches the `POSITION_OPEN` or `partially_closed`/`closed` paper stages.
  - The intermediate bridge stages (`pending`, `rejected_risk`, etc.) must be rendered with their specific reason-code indicators.
- [ ] **Signal Trail Header Hierarchy:**
  - The overall trail-pill must match the `overall` field in the API schema (`OPEN`, `CLOSED_TP`, `CLOSED_SL`, `CLOSED_MANUAL`, `PAPER_REJECTED`, `ENTRY_DISABLED`, `BRIDGE_REJECTED`, `SOURCE_SKIPPED`, `EXPIRED`, `PENDING_ENTRY`, `NOT_APPROVED`).
  - Colors/tones must match the severity scale defined in the backend state machine:
    - **Positive (Green):** `OPEN`, `CLOSED_TP`
    - **Neutral/Warning (Yellow/Orange):** `PENDING_ENTRY`, `REQUIRES_REVIEW`, `PARTIALLY_CLOSED`
    - **Negative/Block (Red/Gray):** `ENTRY_DISABLED`, `BRIDGE_REJECTED`, `PAPER_REJECTED`, `SOURCE_SKIPPED`, `EXPIRED`, `NOT_APPROVED`

---

## 3. Portfolio & PnL Attribution

- [ ] **Source Attribution:**
  - Portfolio PnL must remain source-attributed. The dashboard must show exactly whether PnL was generated by the autonomous trading loop or by the telegram premium channel signal path.
- [ ] **Scale-Invariance on Exit:**
  - Completion PnL must require an entry fill on the *same scale* as the exit message.
  - If the touch price from the completion event was on a different scale (e.g. 10x deviation), it must show the **REQUIRES_REVIEW** scale review warning and suppress the PnL calculation in the UI.

---

## 4. Visual Layout Rules & Indicators

- [ ] **Scale Warning Indicators:**
  - If `scale_unknown` is `true`, a yellow alert icon and message "Scale Unknown — Falling back to per-tick detection" must be displayed next to the signal parameters.
- [ ] **Audit Trail Timelines:**
  - The pipeline lanes in the `PremiumSignalTrail` component (Parsed → Envelope → Approved → Bridge → Paper → Closed) must draw clear connections:
    - **Gray Check:** Step not reached or not evaluated.
    - **Green Check:** Step succeeded with timestamp.
    - **Red Cross:** Step failed or was rejected, showing the reason code (e.g., `max_open_positions`, `entry_mode_disabled`).
- [ ] **Next Action Hints:**
  - Action buttons must reflect the true potential recovery path:
    - `manual_fill` for signals that are `NOT_APPROVED`.
    - `review_reason` for `BRIDGE_REJECTED`.
    - `review_scale` for `PAPER_REJECTED` or scale discrepancies.
    - `wait_or_reprocess` for `PENDING_ENTRY`.
